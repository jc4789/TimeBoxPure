# Plan-implimentation-guidelines

## Purpose

実装計画を固定命令ではなく、更新可能な作業台帳として使う。作業内容を切り替える前に計画、ユーザー要求、現在のコード状態を再確認し、完了した作業、残課題を計画へ反映する。計画が確認済みの事実と矛盾する場合は盲従せず、根拠と影響を記録して計画を修正してから続行する。

## Trigger When

Use this skill when:

- ユーザーの要求と関連する計画全体を読む。
- 現在のコード、設定、依存関係を確認する。
- 計画項目を、未着手、進行中、完了、保留、変更のいずれかとして把握する。
- 最初に実行する項目と、その完了条件を特定する。

## 実行ループ 
一度に一つのまとまった計画項目へ取り組む。

作業対象を別の項目へ切り替える前に、次を行う。

1. 計画の関連部分と未完了項目を再確認する。
2. 実際に変更した内容を計画へ反映する。
3. 変更したファイル、重要な判断、残課題を簡潔に記録する。
4. 次の作業が元の目的と現在のコード状態に一致していることを確認する。

ファイルを一つ編集するたびに計画を更新する必要はない。意味のある作業単位が完了した時、作業対象を変更する時、または前提が変わった時に更新する。

計画が書き込み可能な追跡文書として用意されている場合は、その文書へ状態と実績を追記する。読み取り専用の資料、ユーザーが提供した原文、または変更対象外の文書である場合は、原文を改変せず、同じ項目構造の実行台帳を作業状態として維持する。

## 計画が間違っている場合 

確認済みのコード、テスト結果、依存関係、またはユーザーの要求と計画が矛盾する場合、計画へ盲従しない。

矛盾を発見したら次を行う。

1. 矛盾している計画項目を特定する。
2. 判断の根拠となるファイル、コード、設定、または実行結果を確認する。
3. 計画どおりに進めた場合の影響を判断する。
4. 元の目的と範囲を維持できる最小限の修正を計画へ記録する。
5. 修正後の計画に従って作業を続ける。

明確なコード上の証拠があり、変更が元の目的と範囲を保つ場合は、計画を修正して続行してよい。

次のいずれかを変更する必要がある場合は、勝手に計画を書き換えて実装を続けない。

・　ユーザーが明示した要求
・　公開されている動作またはインターフェース
・　互換性要件
・　主要な設計方針
・　作業範囲
・　計画内の重要な成果物

その場合は、該当項目を保留として記録し、矛盾、根拠、必要な判断を報告する。


## 根拠のない断定を防ぐ  

確認していないファイル、型、関数、設定、依存関係、テスト結果を事実として扱わない。

実装判断に必要な情報は、関連するコードまたは実行結果から確認する。確認できない内容は推測または未確認として記録する。

次の行為をしない。

- 存在を確認していないコードを前提にする
- 実行していない検証を成功扱いにする
- 一部の変更だけで計画項目全体を完了扱いにする
- 新しく発見した問題へ理由なく作業範囲を広げる
- 計画から外れた変更を記録せずに行う

計画外の問題を発見した場合、現在の目的に必要なら計画へ追加する。必要でなければ追跡項目として記録し、現在の作業範囲には含めない。  

## 完了判定 

計画項目は、次の条件を満たした場合だけ完了とする。

- 必要な変更が実際に存在する
- 計画に定められた成果を満たしている
- 適切な検証を実行している
- 検証できなかった内容と理由を記録している
- 残っている問題または副作用を記録している

作業終了時に、計画、実際の差分、検証結果を照合する。

最終報告には次を含める。

- 完了した計画項目
- 変更または省略した計画項目と理由

- 未完了、保留、未確認の項目
