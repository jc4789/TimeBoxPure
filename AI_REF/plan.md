# TimeBoxVibe — Magic Circle & Demoscene Plan

**Status:** Locked, awaiting final answers to open questions before implementation.
**Date:** 2026-06-26
**Engine target:** Kotlin Multiplatform commonMain, Android host.

---

## 0. Executive Summary

The current magic circle (16 layers, integer-snap raster, wall-clock phase) is functional but has 4 visible issues from the screenshots:

1. **Jittery animation** — `getEpochMillis() % 48000L` wraps every 48 seconds; all 7 rotation phases snap back simultaneously.
2. **Slow / stuttery movement** — frame timing inconsistency + low per-layer rates (0.10–1.0 ratios in a 48s period).
3. **Ghosted / doubled lines** — the outer pentagram, guide line, inner pentagram, and octagram squares all overlap the same pixel band.
4. **Palette underuse** — only 4-5 of 16 palette indices; missing the "outer dot ring" decoration.

The plan delivers:

- **Engine foundation:** `FrameClock` (monotonic frame counter), `Wave` (sin/cos oscillator), `PerlinNoise` (1D/2D/FBM, seeded), `IkChain2D` (FABRIK solver).
- **Renderer rewrite:** 9 explicit non-overlapping bands, real PC-98 aesthetic preserved (integer-snap kept, LUT kept).
- **Demoscene effects:** sin/cos waves on 7 layers, Perlin-driven rune drift and optional background nebula (off by default), 1 FABRIK chain for the yin-yang comet trail.
- **Settings:** "Background Nebula" toggle (default OFF), "Demoscene Effects" master toggle (default ON).
- **No new dependencies, no asset pipeline, no new framework.** All ~400 new lines are pure Kotlin, allocation-free in hot paths, deterministic.

**FPU / integer-snap rule:** **kept as-is.** The integer-snap is what gives the PC-98 identity; the 1-pixel jitter is the aesthetic, not a bug. FPU is already used for animation phases (smooth motion). The LUT (1024 entries, 0.35°/step) is fast and accurate enough. The 1px jitter is sub-LUT-resolution and indistinguishable from sub-pixel rendering. Relaxing the rule would lose the PC-98 look without measurable visual gain.

**SDF / ray marching:** **out of scope for this pass, documented as future.** Both are CPU-bound (each pixel does 5-10 distance evaluations = ~40M ops/frame at 800×480). They would fundamentally change the aesthetic from "PC-98 hard-edged" to "modern anti-aliased demo", which conflicts with the timeboxing identity. They could be a future "Modern Mode" preset. SDF is the most useful of the two for glyph anti-aliasing at small scales.

## 2. The 3 Demoscene Techniques — Confirmed

User confirmed: **all 3** (sin/cos + Perlin + FABRIK).

### 2.1 sin/cos wave functions (Tier S)

**Implementation:** `Wave` class in `commonMain` — pre-allocated per-layer instance, `Wave.update(dt)` called per frame from `SceneManager.update()`.

**7 instances planned:**
| # | Target | Freq (Hz) | Amplitude | Effect |
|---|---|---|---|---|
| 1 | Rune band kanji sway | 0.3 | 0.5px y-offset | Each glyph sways with phase stagger |
| 2 | Pentagram breath | 0.5 | ±2.5% radius | Subtle in/out |
| 3 | Outer bead ring heartbeat | 0.7 | +0–40% strokeWidth | Visible pulse |
| 4 | Inner bead ring heartbeat | 0.7 (π offset) | +0–40% strokeWidth | Double-beat |
| 5 | Yin-yang core wobble | 1.2 | ±4% scale | Tight wobble |
| 6 | Decoration dot pulse | 0.4 | +0–50% size | Cardinal gold dots breathe |
| 7 | Sector kanji swing | 0.6 | ±3% radius | 5 sector kanji swing in/out |

### 2.2 Perlin Noise (Tier A)

**Implementation:** `PerlinNoise` object in `commonMain` — 256-entry permutation table, doubled to 512 for wraparound. Generated by `gen_lut.py` (new `--kind perm` flag) and emitted via `kotlin_const_emit.py`. Seed = 42 (fixed, deterministic).

**3 applications planned:**
1. **Rune band drift** — `perlin1D(seed=42, t=phase + idx*0.1) * 0.5°` per glyph angle perturbation.
2. **Background nebula** — `fbm(seed=42, x, y, 2 octaves) * 0.5` modulating BG color index between `BG` (0) and `BG_ALT` (1). Sampled at 8×8 grid (cheap, ~30 ops total per frame). **OFF by default** (settings toggle).
3. *Timer text ripple — **SKIPPED**. Digits must stay crisp.*

**Why defer the timer text:** the mmss readout is a productivity-critical UI element. Perlin y-ripple would hurt readability. Spell-card aesthetic does not require wiggling digits.

### 2.3 FABRIK IK chains (Tier B)

**Implementation:** `IkChain2D` class in `commonMain` — 6 linked points, FABRIK solver (forward+backward pass, 6 iterations). ~10 ops per segment per iteration = ~120 ops/frame.

**1 application planned:**
- **Yin-yang comet trail** — 6 chain links, each rendered as a small dot with decreasing alpha (0xFF → 0x40) and size (3px → 1px). Target follows the rotating yin-yang with a per-link delay. Looks like a comet tail.

**Why only 1 use:** other IK demos (pentagram wobble, elastic rune ring, square stretch) are cool but distracting in a timer. The comet tail is the highest-impact-per-cost.

## 3. FPU / Integer-Snap Discussion

### 3.1 Current state

- `EngineCanvas.setPixel(x, y, color)` does `roundToInt()` on x and y.
- All `drawLine`, `drawRect`, `drawCircle`, `drawGlyph` etc. integer-snap.
- LUT (`FastMath.TABLE_SIZE = 1024`) for `sin`/`cos` — 0.35°/step.
- No `kotlin.math.sin/cos` in hot path.
- Animation phases (e.g., `phase * 360 * ratio`) are floats.

### 3.2 What "relaxing the integer-snap rule" would actually do

| Path | Resolution | Visual | Cost |
|---|---|---|---|
| **Current (LUT + integer-snap)** | 0.35° angular, 1px spatial | PC-98 hard-edged, "jittery" 1px stair-step in motion | Cheapest (LUT + round) |
| **Drop LUT, use FPU sin/cos + integer-snap** | Sub-degree angular, 1px spatial | Identical visual to current, since integer-snap dominates | Slower (FPU call vs LUT lookup), no visual gain |
| **LUT + sub-pixel raster (anti-aliased)** | 0.35° angular, 0.01px spatial | Smooth, modern, no PC-98 aesthetic | ~4× cost (4 sub-pixel samples per output pixel) |
| **FPU + sub-pixel raster** | Sub-degree, 0.01px | Smoothest, fully modern | ~4× cost |

### 3.3 Verdict

**Keep the integer-snap rule.** Rationale:

1. **The integer-snap is the PC-98 identity.** Hard-edged pixels, no anti-aliasing. Removing it loses the look.
2. **The "jitter" is sub-LUT-resolution and visually identical to sub-pixel rendering** at typical radii (200px). Below human perception for most users.
3. **Animation phases are already floats.** The "jitter" comes from the integer-snap at the final pixel, not from the math. FPU won't help.
4. **Cost is non-trivial.** Sub-pixel rendering is ~4× the cost. Mobile GPU is happy with current 6,400 setPixel/frame; 4× = 25,600 setPixel/frame may push frame time over budget.
5. **For demoscene effects (waves, Perlin, IK), FPU is already the right tool** — the math is float, the final raster is integer-snap. We're not bottlenecked by FPU.

### 3.4 The only case where FPU matters

For the **sin/cos in `Wave.value()`**, the LUT is the right tool (1024 entries covers 0.35°/step, which is sub-pixel at typical radii). No need to switch.

For the **Perlin noise**, the permutation table + lerp is fast and deterministic. No FPU noise calls needed.

For the **FABRIK chain**, the math is straight addition/multiplication/sqrt — all FPU, no trig.

**Conclusion: the FPU is already used where it matters (intermediate math, phases). The integer-snap is preserved at the final raster. No rule changes needed.**

## 4. SDF / Ray Marching Thoughts

### 4.1 What they are

- **SDF (Signed Distance Field):** for each pixel, compute the distance to a shape. Useful for: smooth anti-aliased shapes, "glow" or "halo" effects, smooth dot rings, smooth text.
- **Ray marching:** shoot a ray through the scene, sample SDF at each step, accumulate color. Useful for: 3D-feel effects, refractive shapes, volumetric effects, soft shadows.

### 4.2 Cost analysis for our framebuffer

For a 800×480 framebuffer:
- SDF per pixel: 5-10 distance evaluations × 4 channels = 20-40 ops/pixel = **16-32M ops/frame**
- Ray marching per pixel: 20-30 steps × SDF eval = 100-300 ops/pixel = **80-240M ops/frame**

For comparison, current magic circle: ~6,400 setPixel/frame ≈ 0.4% of one CPU core on a modern phone.

SDF would be **5-10× the cost** of the current circle. Ray marching would be **25-60× the cost**. Both are doable on a high-end phone but **not on a budget Android device**.

### 4.3 Aesthetic impact

SDF + ray marching would fundamentally change the visual style:
- **Anti-aliased** (no hard edges) — conflicts with PC-98 aesthetic
- **Smooth shapes** (no pixel stair-step) — conflicts with PC-98 aesthetic
- **Volumetric / 3D feel** — closer to modern demoscene than PC-98
- **Soft glow / halo** — would actually look amazing on the alarm flash

### 4.4 Verdict

**Skip for this pass.** Documented as future work.

- **SDF alone** (no ray marching) is the most useful of the two. Could be used for:
  - Smooth glyph rendering at small scales (sub-cell positioning)
  - "Glow" halos around the magic circle at alarm time
  - Soft dot rings (instead of integer-snapped filled circles)
  - Smooth bead ring (the 60 outer beads rendered as soft glowing dots)
  - Cost: ~10-20M ops/frame (acceptable on modern phones, too heavy on budget devices)
  - **Could be a "Modern Mode" preset that the user can toggle**

- **Ray marching** is overkill. Would only be used for:
  - Volumetric alarm (smoke/energy emanating from the center)
  - 3D-feel "folding" effects
  - **Skip entirely for now.**

### 4.5 Implementation path if we add SDF later

1. Add `SdfShape` sealed class with `circle`, `line`, `rect`, `polygon` variants.
2. Add `sdfSample(sdf: SdfShape, x: Float, y: Float): Float` — returns signed distance (negative inside, positive outside).
3. Add `SdfRenderer` object that walks the framebuffer and renders a list of SDFs.
4. For the alarm "glow" effect: add a radial SDF around the magic circle center, color falls off as `1 / (1 + dist * dist)`.
5. Settings toggle: "Modern SDF Mode" (default OFF, since it costs CPU).

## 5. Engine Upgrades — 4 New Files

All in `shared-engine/src/commonMain/kotlin/com/example/timeboxvibe/engine/core/`.

### 5.1 `FrameClock.kt` (~30 lines)

```kotlin
package com.example.timeboxvibe.engine.core

object FrameClock {
    private var frameCount: Long = 0L

    fun tick() { frameCount++ }
    fun reset() { frameCount = 0L }
    fun frame(): Long = frameCount

    fun phase(periodFrames: Long): Float =
        ((frameCount % periodFrames).toFloat() / periodFrames.toFloat())

    fun rotation(degreesPerFrame: Float): Float {
        val raw = (frameCount.toDouble() * degreesPerFrame) % 360.0
        return raw.toFloat()
    }

    fun seconds(frameRate: Float): Float = (frameCount.toFloat() / frameRate)
}
```

- `tick()` called from `SceneManager.update()` at top of each frame.
- `phase(N)` for wrap-friendly cycles (rune band 60s, alarm 0.25s, etc.).
- `rotation(rate)` for continuous rotation in degrees/frame, frame-rate independent.
- `seconds(rate)` for time-since-start at a given frame rate.

### 5.2 `Wave.kt` (~30 lines)

```kotlin
package com.example.timeboxvibe.engine.core

class Wave {
    var phase: Float = 0f
    var amplitude: Float = 1f
    var frequency: Float = 1f
    var baseValue: Float = 0f
    var phaseOffset: Float = 0f

    fun update(dt: Float) { phase += dt * frequency }
    fun value(): Float = baseValue + amplitude * sin((phase + phaseOffset) * 2f * 3.14159265f)
    fun valueNorm(): Float = 0.5f + 0.5f * value()
}
```

- 7 instances pre-allocated in `ActiveTimerScene` init.
- `update(dt)` called per frame from `SceneManager.update()`.
- `value()` called per-frame from the renderer.
